* Introduction
        #+ATTR_LATEX: :options []
        #+BEGIN_definition
        The *all-prefix-sums* operation takes a binary associative operator \(\oplus\), and an ordered set of
        \(n\) elements
        \begin{equation*}
        [a_0,a_1,\dots,a_{n-1}]
        \end{equation*}
        and returns the ordered set
        \begin{equation*}
        [a_0, (a_0\oplus a_1),\dots,(a_0\oplus a_2\oplus\cdots\oplus a_{n-1})]
        \end{equation*}
        #+END_definition

        Uses:
        1. To lexically compare strings of characters. For example, to determine that ~strategy~ should appear
           before ~stratification~ in a dictionary
        2. To add multi precision numbers. These are numbers that cannot be represented in a single machine word.
        3. To evaluate polynomials
        4. To solve recurrences. For example, to solve the recurrences \(x_i=a_ix_{i-1}+b_ix_{i-2}\) and
           \(x_i=a_i+b_i/x_{i-1}\)
        5. To implement radix sort
        6. To implement quicksort
        7. To solve tridiagonal linear systems
        8. To delete marked elements from an array
        9. To dynamically allocate processors
        10. To perform lexical analysis. For example, to parse a program into tokens.
        11. To search for regular expressions. For example, to implement the UNIX grep program. 
        12. To implement some tree operations. For example, to find the depth of every vertex in a tree
        13. To label components in two dimensional images.

        The *scan* operation is a vector all-prefix-sums operation

        The *prescan* operation takes a binary associative operator \(\oplus\) with identity \(I\), and a vector
        of \(n\) elements
        \begin{equation*}
        [a_0,\dots,a_{n-1}]
        \end{equation*}
        and returns the vector
        \begin{equation*}
        [I,a_0,(a_0\oplus a_1),\dots,(a_0\oplus a_1\oplus\cdots\oplus a_{n-2})]
        \end{equation*}
        #+END_definition

* Implementation
        This section describes an algorithm for calculating the scan operation in parallel. For \(p\)
        processors and a vector of length \(n\) on an EREW [[https://en.wikipedia.org/wiki/Parallel_RAM][PRAM]], the algorithm has a time complexity of
        \(O(n/p+\log p)\). The algorithm is simple and well suited for direct implementation in hardware.
        Chapter 4 shows how the time of the scan operation with certain operators can be reduced to
        \(O(n/p+\log p/\log\log p)\) on a CREW PRAM.
        
        Before describing the scan operation, we consider a simpler problem, that of generating only the final
        element of the scan. We call this the *reduce* operation.

        #+ATTR_LATEX: :options []
        #+BEGIN_definition
        The *reduce* operation takes a binary associative operator \(\oplus\) with identity \(i\), and an
        ordered set \([a_0,\dots,a_{n-1}]\) of \(n\) elements, and returns the value \(a_0\oplus\dots\oplus a_{n-1}\)
        #+END_definition

        A balanced binary tree can be used to implement the reduce operation by laying the tree over the
        values, and using \(\oplus\) to sum pairs at each vertex (see Figure [[ref:1.2]]a). The correctness of the
        result relies on \(\oplus\) being associative. The operator, however, does not need to be commutative
        since the order of the operands is maintained.

        On an EREW PRAM, each level of the tree can be executed in parallel, so the implementation can step
        from the leaves to the root of the tree (see Figure [[ref:1.2]]b); we call this an *up-sweep*. Since the
        tree is of depth \(\ceil{\log n}\), and one processor is needed for every pair of elements, the
        algorithm requires \(O(\log n)\) time and \(n/2\) processors.

        #+ATTR_LATEX: :width .7\textwidth :float nil
        #+NAME: f1.2
        #+CAPTION: 
        [[../images/papers/320.png]]

        #+ATTR_LATEX: :width .7\textwidth :float nil
        #+NAME: f1.3
        #+CAPTION: 
        [[../images/papers/321.png]]

        If we assume a fixed number of processors \(p\), with \(n>p\), then each processor can sum an \(n/p\)
        section of the vector to generate a processor sum; the tree technique can then be used to reduce the
        processor sums (see Figure [[ref:f1.3]]). The time taken to generate the processor sums is \(\ceil{n/p}\),
        so the total time required on an EREW PRAM is:
        \begin{equation*}
        T_R(n,p)=\ceil{n/p}+\ceil{\log p}=O(n/p+\log p)
        \end{equation*}

        
        When \(n/p\ge\log p\) the complexity \(O(n/p)\).

        We now return to the scan operation. We actually show how to implement the prescan operation; the scan
        is then determined by shifting the result and putting the sum at the end. If we look at the tree
        generated by the reduce operation, it contains many partial sums over regions of the vector. It turns
        out that these partial sums can be used to generate all the prefix sums. This requires executing
        another sweep of the tree with one step per level, but this time starting at the root and going to the
        leaves (a down-sweep). Initially, the identity element is inserted at the root of the tree. On each
        step, each vertex at the current level passes to its left child its own value, and it passes to its
        right child, \(\oplus\) applied to the value from the left child from the up-sweep and its own value

        #+ATTR_LATEX: :width .7\textwidth :float nil
        #+NAME: f1.4
        #+CAPTION: 
        [[../images/papers/322.png]]

        Let's consider why the down-sweep works. We say that vertex \(x\) *precedes* vertex \(y\) if \(x\)
        appears before \(y\) in the preorder (root, left, right) traversal of the tree 

        #+ATTR_LATEX: :options []
        #+BEGIN_theorem
        [[label:thm1.1]]
        After a complete down-sweep, each vertex of the tree contains the sum of all the leaf values that
        precedes it
        #+END_theorem

        #+BEGIN_proof
        Induction on the root: if a parent has the correct sum, both children must have the correct sum. The
        root has no element preceding it, so its value is correctly the identity element.

        Consider Figure [[ref:f1.5]]. The left child of any vertex has exactly the same leaves preceding it as the
        vertex itself (the leaves in region A in the figure). This is because the preorder traversal always
        visits the left child of a vertex immediately after the vertex. By the induction hypothesis, the
        parent has the correct sum, so it need only copy this sum to the left child.

        The right child of any vertex has two sets of leaves preceding it, the leaves preceding the parent
        (region A), and the leaves at or below the left child (region B). Therefore, by adding the parent’s
        down-sweep value, which is correct by the induction hypothesis, and the left-child’s up-sweep value,
        the right-child will contain the sum of all the leaves preceding it. 

        #+END_proof

        #+ATTR_LATEX: :width .7\textwidth :float nil
        #+NAME: f1.5
        #+CAPTION: 
        [[../images/papers/323.png]]

        On an EREW PRAM, the running time is \(2\ceil{\log n}\)

        If we assume a fixed number of processors \(p\), with \(n>p\). Each processor first sums an \(n/p\)
        section of the vector to generate a processor sum, the tree technique is then used to prescan the
        processor sums. The results of the prescan of the processor sums are used as an offset for each
        processor to prescan within its \(n/p\) section (see Figure [[ref:f1.6]]). The time complexity of the
        algorithm is:
        \begin{equation*}
        T_S(n,p)=2(\ceil{n/p}+\ceil{\log p})=O(n/p+\log p)
        \end{equation*}

* Line-of-Sight and Radix-Sort
        Line-of-sight: given a terrain map in the form of a grid of altitudes and an observation point \(X\) on
        the grid, find which points are visible along a ray originating at the observation point. A point on a
        ray is visible if and only if no other point between it and the\ observation point has a greater
        vertical angle. 

        #+ATTR_LATEX: :width .7\textwidth :float nil
        #+NAME: f1.7
        #+CAPTION: 
        [[../images/papers/343.png]]

        To find if any previous point has a greater angle, the altitude of each point along the ray is placed
        in a vector (the *altitude vector*). These altitudes are then converted to angles and placed in the
        *angle vector*. A prescan using the operator \(\maximum\) (\(\maxprescan\)) is then executed on the
        *angle vector*, which returns to each point the maximum previous angle. To test for visibility, each
        point only needs to compare its angle to the result of the \(\maxprescan\).

        For \(n\) points on a ray, the complexity of the algorithm is \(T_S(n,p)=O(n/p+\log n)\) on an EREW PRAM.


        Now consider /radix sort/ algorithm. The algorithm loops over the bits of the keys, starting at the
        lowest bit, executing a \(\textsf{split}\) operation on each iteration (assume all the keys have the
        same number of bits). The \(\ssplit\) operation packs the keys with a 0 in the corresponding bit to the
        bottom of a vector, and packs the key with a 1 in the bit to the top of the same vector.

        #+ATTR_LATEX: :width .7\textwidth :float nil
        #+NAME: f1.8
        #+CAPTION: 
        [[../images/papers/344.png]]

        We now consider how to \(\ssplit\) can be implemented using a scan. To determine the new indices for
        elements with a 0 in the bit, we invert the flags and execute a prescan with integer addition. To
        determine the new indices of elements with a 1 in the bit, we execute a \(+\text{-}\scan\) in reverse
        order (starting at the top of the vector) and subtract the results from the length of the vector n.

        #+ATTR_LATEX: :width .7\textwidth :float nil
        #+NAME: f1.9
        #+CAPTION: 
        [[../images/papers/345.png]]

        \(\ssplit\): \(O(n/p+\log p)\)

* Recurrence Equations
        A recurrence is a set of equations of the form
        \begin{equation*}
        x_i=f_i(x_{i-1},x_{i-2},\dots,x_{i-m}),\quad m\le i<n
        \end{equation*}
        along with a set of initial values \(x_0,\dots,x_{m-1}\)

        The scan operation is the special case of a recurrence of the form
        \begin{equation}
        \label{1.4}
        x_i=
        \begin{cases}
        a_0&i=0\\
        x_{i-1}\oplus a_i&0<i<n
        \end{cases}
        \end{equation}

        
** First-Order Recurrences
        we consider
        \begin{equation}
        \label{1.5}
        x_i=
        \begin{cases}
        b_0&i=0\\
        (x_{i-1}\otimes a_i)\oplus b_i&0<i<n
        \end{cases}
        \end{equation}
        where \(a_i\) and \(b_i\) are sets of \(n\) arbitrary constants and \(\oplus,\otimes\) are arbitrary
        binary operator satisfying:
        1. \(\oplus\) is associative
        2. \(\otimes\) is semiassociative: there exists a binary associative operator \(\odot\) s.t.
           \((a\otimes b)\otimes c=a\otimes(b\odot c)\)
        3. \(\otimes\) distributes over \(\oplus\): \(a\otimes(b\oplus c)=(a\otimes b)\oplus(a\otimes c)\)

        \(\odot\) is called the *companion operator* of \(\otimes\). If \(\otimes\) is fully associative, then
        \(\odot\) and \(\otimes\) are equivalent

        We now show how eqref:1.5 can be reduced to eqref:1.4. Consider the set of pairs
        \begin{equation*}
        c_i=[a_i,b_i]
        \end{equation*}
        and define a new binary operator \(\bullet\) as follows
        \begin{equation*}
        c_i\bullet c_j\equiv[c_{i,a}\odot c_{j,a},(c_{i,b}\otimes c_{j,a})\oplus c_{j,b}]
        \end{equation*}
        where \(c_{i,a}\) and \(c_{j,b}\) are the first and second elements of \(c_i\) respectively

        \(\bullet\) is associative
        \begin{align*}
        &(c_i\bullet c_j)\bullet c_k=\\
        &\quad =[c_{i,a}\odot c_{j,a},(c_{i,b}\otimes c_{j,a})\oplus c_{j,b}]\bullet c_k\\
        &\quad=[(c_{i,a}\odot c_{j,a})\odot c_{k,a},(((c_{i,b}\otimes c_{j,a})\oplus c_{j,b})\otimes c_{k,a})\oplus c_{k,b}]\\
        &\quad=[c_{i,a}\odot(c_{j,a}\odot c_{k,a}),((c_{i,b}\otimes c_{j,a})\otimes c_{k,a})\oplus((c_{j,b}\otimes c_{k,a})
        \oplus c_{k,b})]\\
        &\quad=[c_{i,a}\odot(c_{j,a}\odot c_{k,a}),(c_{i,b}\otimes(c_{j,a}\odot c_{k,a}))\oplus((c_{j,b}\otimes c_{k,a})
        \oplus c_{k,b})]\\
        &\quad=c_i\bullet(c_j\bullet c_k)
        \end{align*}

        We now define the ordered set \(s_i=[y_i,x_i]\), where the \(y_i\) obey the recurrence
        \begin{equation*}
        y_i=
        \begin{cases}
        a_0&i=0\\
        y_{i-1}\odot a_i&0<i<n
        \end{cases}
        \end{equation*}
        and the \(x_i\) are from eqref:1.5. We get
        \begin{align*}
        s_0&=[y_0,x_0]\\
        &=[a_0,b_0]\\
        &=c_0\\
        s_i&=[y_i,x_i]\\
        &=[y_{i-1}\odot a_i,(x_{i-1}\otimes a_i)\oplus b_i]\\
        &=[y_{i-1}\odot c_{i,a},(x_{i-1}\otimes c_{i,a})\oplus c_{i,b}]\\
        &=[y_{i-1},x_{i-1}]\bullet c_i\\
        &=s_{i-1}\bullet c_i
        \end{align*}

        This allows us to use the scan with operator \(\bullet\) to solve any recurrence of the form eqref:1.5
        on an EREW PRAM in time:
        \begin{equation*}
        (T_{\odot}+T_{\otimes}+T_{\oplus})T_S(n,p)=2(T_{\odot}+T_{\otimes}+T_{\oplus})(n/p+\log p)
        \end{equation*}

        
** Higher Order Recurrences
        Now consider
        \begin{equation*}
        x_i=
        \begin{cases}
        b_i&0\le i<m\\
        (x_{i-1}\otimes a_{i,1})\oplus\cdots\oplus(x_{i-m}\otimes a_{i,m})\oplus b_i&m\le i<n
        \end{cases}
        \end{equation*}
        with same restrictions as eqref:1.5

        Define
        \begin{equation*}
        s_i=
        \begin{bmatrix}
        x_i&\cdots&x_{i-m+1}
        \end{bmatrix}
        \end{equation*}

        
* Segmented Scans
        This section shows how the vector operated on by a scan can be broken into segments with flags so that
        the scan starts again at each segment boundary

        #+ATTR_LATEX: :width .7\textwidth :float nil
        #+NAME: f1.10
        #+CAPTION: 
        [[../images/papers/346.png]]

        Each of these scans takes two vectors of values: a *data vector* and a *flag vector*. The segmented scan
        operations present a convenient way to execute a scan independently over many sets of values. The next
        section shows how the segmented scans can be used to execute a parallel quicksort

        The segmented scans satisfy the recurrence
        \begin{equation*}
        x_i=
        \begin{cases}
        a_0&i=0\\
        \begin{cases}
        a_i&f_i=1\\
        x_{i-1}\oplus a_i&f_i=0
        \end{cases}&0<i<n
        \end{cases}
        \end{equation*}
        where \(\oplus\) is the original associative scan operator. If \(\oplus\) has identity \(I_{\oplus}\),
        then we have
        \begin{equation*}
        x_i=
        \begin{cases}
       a_0&i=0\\
        (x_{i-1}\times_sf_i)\oplus a_i&0<i<n 
        \end{cases}
        \end{equation*}
        where
        \begin{equation*}
        x\times_sf=
        \begin{cases}
        I_{\oplus}&f=1\\
        x&f=0
        \end{cases}
        \end{equation*}
        This is in the form eqref:1.5 and \(\times_s\) is semiassociative with logical *or* as the companion
        operator. Therefore we can execute the segmented scans in time
        \begin{equation*}
        (T_{or}+T_{\times_s}+T_{\oplus})T_S(n,p)
        \end{equation*}

** Example: Quicksort
        Consider a parallel version of quicksort. Similar to the standard sequential version, the parallel
        version picks one of the keys as a pivot value, splits the keys into three sets—keys lesser, equal and
        greater than the pivot—and recurses on each set.

        Complexity: \(O(T_S(n,p)\log n)=O(\frac{n}{p}\log n+\log^2n)\)

        #+ATTR_LATEX: :width .7\textwidth :float nil
        #+NAME: f1.11
        #+CAPTION: 
        [[../images/papers/347.png]]

        1. Check if sorted: reduce with logical *and*
        2. Within each segment, pick a pivot and distribute it to the other elements.

           If we pick the first element as a pivot, we can use a segmented scan with the binary operator
           \(\textsf{copy}\), which returns the first of its arguments
           \begin{equation*}
           a\gets\textsf{copy}(a,b)
           \end{equation*}
        3. Within each segment, compare each element with the pivot and split based on the result of the
           comparison

           We can use a version of the \(\ssplit\) which splits into three sets instead of two, and which is
           segmented. To implement such a segmented \(\ssplit\), we can use a segmented version of the
           +-\(\scan\) operation to generate indices relative to the beginning of each segment, and we can use
           a segmented \(\copyscan\) to copy the offset of the beginning of each segment across the segment.
           We then add the offset to the segment indices to generate the location to which we permute each
           element. 
        4. Within each segment, insert additional segment flags to separate the split values
