\section{Introduction}
\label{sec:org080add1}
Flash’s main challenge is its limited write endurance; i.e., flash can only be written so many times
before wearing out. Wearout is especially problematic for tiny objects because flash can be read and
written only at multi-KB granularity. For example, writing a 100 B object may require writing a 4 KB
flash page, amplifying bytes written by 40× and rapidly wearing out the flash device. Thus, cost also
argues for minimizing excess bytes written to flash. 

\textbf{The Problem}. Prior flash-cache designs either use too much DRAM or write flash too much.
\begin{itemize}
\item LSM: tracking billions of tiny objects requires a large index, and even a heavily optimized index
needs large amounts of DRAM
\item Set-associative caches operate by hashing objects’ keys into distinct “sets,” much like CPU caches.
Admitting a single small object to the cache requires re-writing an entire set, significantly
amplifying the number of bytes written to the flash device.
\end{itemize}

\begin{center}
\includegraphics[width=.6\textwidth]{../images/papers/304.png}
\captionof{figure}{\label{f1}}
\end{center}

\begin{itemize}
\item To avoid a large DRAM index, Kangaroo organizes the bulk of cache capacity as a set-associative
cache, called \emph{KSet}.
\item To reduce flash writes, Kangaroo places a small (e.g., 5\% of flash) log-structured cache, called
\emph{KLog}, in front of KSet. KLog buffers many objects, looking for objects that map to the same set in
KSet (i.e., hash collisions), so that each flash write to KSet can insert multiple objects.
\end{itemize}

Our insight is that even a small log will yield many hash collisions, so only a small amount of extra
DRAM (for KLog’s index) is needed to significantly reduce flash writes (in KSet).

\subsection{Tiny objects are important and numerous}
\label{sec:org7c5d94a}
Tiny objects are prevalent in many large-scale systems:
\begin{itemize}
\item At Facebook, small objects are prevalent in the social graph. For example, the average social-graph
edge size is under 100 B. Across edges, nodes, and other objects, the average object size is less
than 700 B. This has led to the development of a dedicated flash caching system for small objects.
\item At Twitter, tweets are limited to 280 B, and the average tweet is less than 33 characters. Due to
the massive and growing number of tweets, Twitter seeks a cost-effective caching solution.
\item At Microsoft Azure, a growing use case is processing updates from sensor data, such as from IoT
devices in Azure Streaming Analytics. Before an update can be processed (e.g., to trigger a
real-time action), the server must fetch metadata (the sensor’s unit of measurement, geolocation,
owner, etc.) with an average size of 300 B. For efficiency and availability, it caches the most
popular metadata. Another use case arises in search advertising, where Azure caches predictions and
other results
\end{itemize}
\subsection{Caching tiny objects in flash is hard}
\label{sec:orgeba8bb7}
DRAM is expensive

Challenges:
\begin{itemize}
\item flash has \emph{limited write endurance}, which means there is a limit on the number of writes before the
flash device wears out and must be replaced
\item \emph{write amplification}: Flash devices suffer from both device-level write amplification and
application-level write amplification
\end{itemize}

\emph{Device-level write amplification} (dlwa) occurs when the flash translation layer (FTL) writes more
flash pages than asked for by the storage application (e.g., file system, database, or cache). Current
flash drives implement the age-old block-storage interface, wherein hosts read and write logical
blocks in a numerical logical-block address (LBA) namespace. Generally, the internal flash-page size
and the external logical-block size are the same, with 4 KB being common, even though the flash device
can only erase pages in much larger (e.g., 256 MB) “erase blocks”. Most dlwa is caused by cleaning
activity that copies live pages elsewhere before an erase block is cleared.

\begin{center}
\includegraphics[width=.6\textwidth]{../images/papers/305.png}
\captionof{figure}{\label{f2}}
\end{center}

Generally speaking, dlwa worsens as more of the raw flash capacity is utilized and as access patterns
consist more of small, random writes. A common approach to reduce dlwa is over-provisioning, i.e.,
only exposing a fraction of the raw flash capacity in the LBA namespace, so that cleaning tends to
find fewer live pages in victim erase blocks. Fig. \ref{f2} shows dlwa vs. utilized capacity for random
4 KB writes to a 1.9 TB flash drive. As expected, dlwa significantly increases as over-provisioning
decreases, from ≈1× at 50\% utilization to ≈10× at 100\% utilization.

\emph{Application-level write amplification} (alwa) occurs when the storage application re-writes some of its
own data as part of its storage management. One form of this is akin to FTL cleaning, such as cleaning
in log-structured file systems or compaction in log-structured merge trees. Another form is caused by
having to write an entire logical block. To write a smaller amount of data, the application must read
the block, install the new data, and then write the entire block. For example, installing 1 KB of new
data into a 4 KB logical block involves rewriting the other 3 KB, giving alwa of 4×. Ideally, the
unmodified data in the block would not have been rewritten.

\section{Kangeroo Overview and Motivation}
\label{sec:org7d83cda}
\begin{center}

\captionof{figure}{\label{f3}}

Lookup:
\begin{description}
\item[{1}] lookups first check the DRAM cache, which is very small (<1\% of capacity).
\item[{2}] If the requested key is not found, requests next check KLog (≈5\% of capacity). KLog maintains a
DRAM index to track objects stored in a circular log on flash.
\item[{3}] If the key is not found in KLog’s index, requests check KSet (≈95\% of capacity). KSet has no
DRAM index; instead, Kangaroo hashes the requested key to find the set (i.e., the LBA(s) on flash)
that might hold the object.
\item[{3a}] If the requested key is not in the small, per-set Bloom filter, the request is a miss.
\item[{3b}] Otherwise, the object is probably on flash, so the request reads the LBA(s) for the given set
and scans for the requested key.
\end{description}
